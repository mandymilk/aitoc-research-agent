# Update Workflow

## Evidence-First Update Rule

Every meaningful update should move through this order:

1. Evidence note
2. Product or company profile
3. Case study narrative
4. Finding or thesis update
5. Published report, if needed

This prevents the repo from becoming opinion-first.

## Pricing Change Workflow

When a product changes pricing:

- Add source-backed evidence.
- Update `data/product_profiles/`.
- Update the corresponding `knowledge_base/products/` note.
- Check if the change affects business-model taxonomy.
- Check if the change affects survival classification.
- Add a finding if the change reveals a broader market pattern.

## New Product Workflow

When adding a new product:

- Create a product profile.
- Create a knowledge-base product note.
- Add at least three evidence notes: official product page, pricing/packaging, and one distribution or adoption source.
- Assign category, owner type, pricing model, and provisional survival classification.
- Create a full case study only if the product changes the research answer.

## Thesis Update Workflow

Create a thesis update when new evidence changes one of:

- The main answer.
- A core hypothesis.
- The confidence level of a hypothesis.
- The survival classification of an anchor case.
- The taxonomy of business models.

Do not update the thesis just because a new article exists.
